The [Pixie editor](/apps/pixie) ships with two complete games, `snake.pixie` and `tetris.pixie`, and their full source sits right there in the editor pane, made to be read and messed with. This module is a tour of the techniques inside them. Open each game in the editor as you read.

## Snake (~170 lines)

Select `snake.pixie` from the dropdown. You should already know the `gameOver` state machine, one-shot input, and use of `POINT`s in a growable array.

**The tail is a queue.** The whole "snake" illusion is four lines on the movement tick:

```
VAR headPos = POINT(x y)
ARR_UNSHIFT tail headPos

IF LEN(tail) > tailLength:
    ARR_POP tail
END
```

Push where the head *was* onto the front; if the tail is now too long, then drop the oldest segment off the back. Eating an apple just does `tailLength = tailLength + 1`. The tail grows because the pop stops happening for one tick. No segment ever "moves" since the body is simply a trail of history.

**Input is buffered, then applied on the tick.** The snake moves only every `tickRate` frames, but polls `BTN` every frame into `nextDx`/`nextDy`, and copies them into `dx`/`dy` on tick frames. ([M1.0.5](/guides/pixie/m01.00.05/) covers why. Presses land between ticks and would otherwise vanish.) Notice also the guard:

```
IF BTN UP:
    IF dy == 0:        // can't reverse into yourself
        nextDx = 0
        nextDy = -16
    END
END
```

You can only turn *perpendicular* to your current motion. That single `IF dy == 0` is what makes reversing into your own neck impossible.

**Implementing difficulty:** `tickRate = MAX(2, 10 - FLOOR(score / 3))` for every 3 apples makes the game one frame faster. This is similar to the clamp-a-ramp idiom as Catch!.

**Things to try:** 
- Change `GRID 256 256` and the wall checks to make a bigger arena.
- Make apples worth more the faster the game is.

## Tetris (~650 lines)

Select `tetris.pixie`.

**The board is a flat array.** 10x20 cells in `NEW_ARRAY(200)`, addressed as `y * 10 + x` everywhere. This is the exact pattern from [M1.0.6](/guides/pixie/m01.00.06/). A cell holds `0` for empty or a color index 1–7.

**Pieces are arrays of offsets.** A tetromino is four `POINT`s relative to a pivot, plus a position:

```
ARR_PUSH blocks POINT(-1 0)
ARR_PUSH blocks POINT(0 0)
ARR_PUSH blocks POINT(1 0)
ARR_PUSH blocks POINT(2 0)    // the I piece
```

Every operation (drawing, collision, locking) is a `FOR_EACH b IN blocks` over those offsets. And rotation takes a simple trick... the 90 degree rotation of offset `(x, y)` is `(-y, x)`:

```
VAR nx = 0 - b.y
VAR ny = b.x
```

Tetris never rotates the real piece and "undoes" it on failure. It builds the rotated offsets in a *temporary* array, tests that every block lands on a legal cell, and only then assigns `blocks = tempRot`. Same for left/right moves... test all four blocks first, move only if all clear. Speculate on a copy and commit if valid.

**Line clears sweep downward.** Finding a full row is a nested loop with an `isFull` flag, and removing it shifts every row above down by one, using `FOR sy FROM ry DOWNTO 1`. This is an example use case of `DOWNTO`.

**The double-tap hard drop** you already met in [M1.0.5](/guides/pixie/m01.00.05/) is implemented as `frameClock` minus `lastDownTap` under 36 frames means slam. The slam itself is the bounded-`FOR`-as-`WHILE` from [M1.0.4](/guides/pixie/m01.00.04/).

**Things to try:**
- Change the palette in the draw section.
- Change the board size to a dimension greater than 10x20.
