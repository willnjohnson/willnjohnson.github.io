## Introduction

You're probably in Dr. Peterson, Dr. Marz, or Dr. Jantz's class looking for a **more-detailed writeup** for building this simulator.

In this guide, I explain the memory hierarchy, how each reference flows through it, and how to build the simulator incrementally by modeling each layer and tracking the right statistics.

***Note:** I implemented this simulator in C in roughly 700 lines of code. You may end up writing more than this, if you are creating helpers, functions for verifying behavior, etc. Start the assignment early!*

You can use [this JS simulator](/apps/memhier) to step through each access.

## Why This Simulator Exists

A real CPU does not talk directly to disk. Instead, it follows a chain of increasingly large and slower storage structures (although this is an oversimplification of an actual memory hierarchy).

```
CPU -> TLB -> Page Table -> L1 Cache -> L2 Cache -> Main Memory -> Disk
```

Each stage is a filter that answers one question:

- Can I resolve the address quickly?
- Is the data already in this smaller, faster structure?
- Do I need to go deeper?

## What a Student Should Model First

Think of the simulator as a sequence of checks for each memory reference:

1. Read the trace entry (R/W + address)
2. Translate the address if virtual memory is enabled
3. Lookup in the data cache
4. If enabled, lookup in L2
5. If still missing, count a main memory or disk reference
6. Update all relevant state and statistics

## Where to Start Coding

Before you touch the main loop, get the boring plumbing working first:

- Parse `trace.config` and load every parameter into named variables. Print them back out immediately, since the assignment expects a config summary before the first trace line runs anyway.
- Precompute the shift and mask values you will reuse on every single reference: `lg2(page_size)`, `lg2(dc_line_size)`, `lg2(l2_line_size)`, and the index masks for each structure. Do this once at startup, not inside the loop.
- Get a bare loop reading `access_type` and `raw_address` from stdin and printing them back before you wire up any cache logic at all.

Once that skeleton runs cleanly, module 2 walks through how to structure the translation and cache stages around it.

## Why the Layers Matter to Code

The simulator is not a single lookup table.

- Each layer has its own state: valid bits, tags, dirty bits, and LRU ages
- Each layer can be enabled or disabled independently
- Each layer may change what happens on a write vs a read

That means the best implementation strategy is:

- Keep each layer separate
- Define a clear API for each layer
- Make the main loop drive the stages in order

## Practical Mental Model

This guide uses three address spaces:

- Trace address: what the input gives you
- Virtual address: used only when virtual memory is enabled
- Physical address: what the caches actually index

A good simulator keeps that separation clear.

## Programming Tips

- Get the config parsing and header printout exactly right before writing any cache logic. The output format is graded line by line, so a misparsed config file will throw off everything downstream.
- Store layer parameters as named variables (`page_size`, `dc_sets`, `dc_linesize`, ...) instead of re-reading the config struct everywhere.

## Common Mistakes

- Mixing raw trace addresses with physical cache lookups
- Updating cache state before translation completes
- Treating L2 and L1 as independent when the assignment requires inclusion
- Forgetting that a write can behave differently in write-through vs write-back

## What to Take Away

After this module, you should be able to answer:

- What is the main pipeline for a single memory reference?
- Why do we need separate stage logic for translation and cache access?
- Why is it useful to model each layer with its own small data structure?
