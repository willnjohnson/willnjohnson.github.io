A good simulator avoids duplicate lookup logic.

## Why Reusable Functions Matter

Every layer performs the same basic task:

- Take an address or page number
- Scan a set or table
- Return whether the entry exists
- Update state on hit or miss

That means a few helper functions can cover multiple layers.

## Core Helper Functions

The implementation can still group repeated behavior even if it computes the results inline.

For example, my implementation uses a shared cache helper that:

- Scans one set for a matching tag
- Bumps LRU ages for every valid line
- Returns an entry pointer on hit
- Selects a victim on miss
- Supports write-through bypass and write-back allocation rules

The page table remains a flat array indexed by VPN, while the cache helper works with physical addresses.

### Typical Helper Signature

A reusable helper can look like this:

```
int cache_lookup(Cache *cache, unsigned long address, char access_type,
                 Entry **entry_out, EvictInfo *evict_out);
```

It should return:

- `HIT` when the requested line is present
- `MISS` when the line is absent but can be allocated
- `MISS_NO_ALLOC` when write-through and no-write-allocate prevent fill

### Translation Logic

Keep the translation code separate from the cache helper.

The translation stage should:

- Compute `vpn` and `offset`
- Consult the TLB if enabled
- Consult the page table if needed
- Allocate a new physical page on page fault
- Update TLB/page table state

### Cache Access Logic

A shared cache helper can be reused for both L1 and L2 if it accepts:

- Base storage pointer
- Number of sets
- Associativity
- Line shift value
- Write policy flag

The helper should return:

- `hit` or `miss`
- Pointer to the matching or victim entry
- Any evicted dirty line info

## The Write Policy Decision

A cache hit on a write needs to know whether it's allowed to set the dirty bit. A cache miss on a write needs to know whether it's allowed to allocate a line at all. Both answers depend on the write policy for that particular cache, and DC and L2 can be configured independently, so don't hardcode one policy's behavior into your generic helper. Pass the policy in as a parameter:

```
if (cache_helper(...) == hit) {
  if (write && write_back) entry->dirty = 1;
} else {
  if (write && no_write_allocate) {
    if (write_through) write_to_lower_level();
  } else {
    allocate_cache_line();
  }
}
```

Work through all four combinations (hit/miss x allocate/no-allocate) on paper for both DC and L2 before you trust your code's output on them.

## Programming Tips

- Avoid one large `switch` that handles TLB, caches, and page table in the same block
- Give each helper a clear return code like `HIT`, `MISS`, or `MISS_NO_ALLOC`
- Keep layer-specific behavior in the caller rather than inside the generic helper
- Pass a struct or pointer for output values instead of many return variables

## Common Mistakes

- Using the same helper for TLB and page table without accounting for different data
- Mixing hit detection with replacement selection
- Writing back dirty lines before invalidating them
- Assuming a cache fill always happens on a write miss

## What to Build First

- A table lookup for the page table
- A set-associative cache lookup function
- An LRU victim chooser
- A wrapper that calls cache lookup and then handles a miss

If these functions are clear, your main loop becomes a short sequence of stage calls.
