The fastest path to a correct simulator is incremental verification.

## Build and Test in Phases

1. Parse the config file
2. Process input addresses
3. Implement the data cache only
4. Add L2 support
5. Add page table translation
6. Add TLB support
7. Add write policies and eviction behavior

If you test after each phase, you will catch bugs earlier.

## Test each subsystem independently

### Cache-only Mode

- Disable virtual memory
- Disable L2
- Treat trace addresses as physical
- Verify that read/write behavior matches your expected access sequence

### L2 Mode

- Enable L2 but keep virtual memory disabled
- Verify that L1 misses can hit L2
- Verify inclusive invalidation if L2 evicts a line

### Page Table Mode

- Enable virtual addresses, disable TLB
- Verify `vpn -> ppn` translation
- Verify page faults when physical frames fill up

### TLB Mode

- Enable TLB and virtual addresses
- Verify TLB hits and misses separately from page table behavior

## Example Validation Checks

- A repeated reference should become a cache hit
- A write to an existing block should set dirty if using write-back
- A page fault should only occur when a new VPN is touched after physical memory is full
- A TLB miss should still succeed if the page table contains the mapping

## Programming Tips

- Write a mini `trace` printer with the intermediate fields
- Compare your output to a known correct pattern
- Test with the smallest possible cache and page table sizes first
- Use a single configuration file format so every mode uses the same parser
- Take advantage of `diff` to check for mismatches, for example:
  - `diff <(./myprogram < trace.dat)   <(./refprogram < trace.dat)`
  - `diff -y --suppress-common-lines   <(./myprogram < trace.dat | head -500)   <(./refprogram < trace.dat | head -500)`

## Common Mistakes

- Assuming a disabled layer still behaves like it is enabled
- Testing too many features at once
- Not resetting state between tests
- Making a bug in one stage look like a bug in another stage

## What To Do

Write a short test harness if possible.

- One reference at a time
- Known expected hit/miss results
- A clear pass/fail pattern

That mindset saves time when the code is complex.
