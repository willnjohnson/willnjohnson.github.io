A trace line is a tiny program input.

- `R` means read
- `W` means write
- The hexadecimal address is the starting byte of the reference

Example:

```
R:0x1a2b3c
W:0x0100
```

Even if the trace is small, the simulator needs to split that address into several pieces.

## Address Breakdown Diagrams

### Virtual address

```
Virtual address (32 bits)
+----------------------+--------+
| VPN (virtual page #) | Offset |
+----------------------+--------+
```

### Physical address

```
Physical address (32 bits)
+----------------------+--------+
| PPN (physical page #) | Offset |
+----------------------+--------+
```

### Cache address

```
Block address
+--------+-------+-------+
| Tag    | Index | Offset|
+--------+-------+-------+
```

The same offset bits appear in both virtual and physical addresses.

## What Matters in Code

Your simulator should compute these values once per reference:

- `offset`: byte location inside a page or cache line
- `vpn`: virtual page number, if virtual addressing is enabled
- `ppn`: physical page number, after translation
- `dc_index`: which cache set the line maps to
- `dc_tag`: the unique part that identifies the block in that set

## How to Compute Them

If sizes are powers of two (which they ought to be), you can use bit operations.

- `offset = address & (page_size - 1)`
- `vpn = address >> page_shift`
- `index = (address >> line_shift) & (nsets - 1)`
- `tag = address >> (line_shift + index_bits)`

That is the core math behind the simulator.

## Output Format and Field Order

When the full hierarchy is enabled, the assignment expects this exact per-reference order:

- Trace address
- Virtual page number (VPN)
- Page offset
- TLB tag
- TLB index
- TLB result
- PT result
- Physical page number
- DC tag
- DC index
- DC result
- L2 tag
- L2 index
- L2 result

A correct `printf` format looks like (refer to `memhier.pdf` that your professor provided to you):

`"%08x %6x %4x %6x %3x %4s %4s %4x %6x %3x %4s %6x %3x %4s"`

If a layer is disabled, leave the irrelevant fields blank.

## Why This Matters

If you get the breakdown wrong, every cache and translation decision will be wrong.

- The same physical address must map to the same cache set every time
- The page offset must be preserved on translation
- The tag must identify the block uniquely within a set

## Programming Tips

- Precompute shift values during configuration
- Avoid repeated arithmetic inside the main loop
- Keep a small helper like `decompose_address()` to return tag/index/offset
- Use unsigned integers for bit masks

## Common Mistakes

- Forgetting to use the same offset for both L1 and page translation
- Reading the trace address as a string and not converting it correctly (trace address are in hexadecimal)
- Combining `vpn` and `ppn` logic when virtual memory is disabled

## Implementation Checklist

- Parse each trace line into access type and numeric address
- Compute offset and page number
- Use virtual/physical selection consistently
- Compute cache tag/index from physical address
- Pass these values into the cache lookup stage
