A simulator is only useful when it tracks what happened.

## What to Count

The assignment expects these counters:

- `dtlb_hits`, `dtlb_misses`
- `pt_hits`, `pt_faults`
- `dc_hits`, `dc_misses`
- `l2_hits`, `l2_misses`
- `reads`, `writes`
- `mem_refs`
- `pt_refs`
- `disk_refs`

## Where Each Counter is Incremented

### TLB Counters

- Increment `dtlb_hits` on a successful TLB lookup
- Increment `dtlb_misses` when the TLB does not contain the VPN

### Page Table Counters

- Increment `pt_refs` on every page table access
- Increment `pt_hits` when the VPN is valid
- Increment `pt_faults` when the VPN is not valid

### Cache Counters

- Increment `dc_hits` when L1 has the data
- Increment `dc_misses` when L1 misses
- Increment `l2_hits` when L2 has the data after an L1 miss
- Increment `l2_misses` when L2 misses

### Memory and Disk Counters

- Increment `mem_refs` only when data must be fetched from lower memory or written back from a dirty line
- Increment `disk_refs` on page fault handling or when writing a dirty page back to disk

## Counter Mapping for One Reference

For a single access, the correct sequence is usually:

- TLB hit: `dtlb_hits++`
- TLB miss: `dtlb_misses++`
- Page table access: `pt_refs++`
- PT valid: `pt_hits++`
- PT fault: `pt_faults++`, then `disk_refs++` if page replacement writes a dirty page
- L1 hit: `dc_hits++`
- L1 miss: `dc_misses++`
- L2 hit after L1 miss: `l2_hits++`
- L2 miss after L1 miss: `l2_misses++`, then `mem_refs++`

## Why Precise Counting Matters

A single reference may hit in one structure and miss in another.

- A TLB miss can still become a page table hit
- An L1 miss can become an L2 hit
- A write-back eviction may generate an extra memory reference

That is why the logic must be explicit.

## Example Tracking Flow

```
read access
  if tlb_enabled:
    if tlb_hit: dtlb_hits++
    else: dtlb_misses++

  pt_refs++
  if pt_valid: pt_hits++
  else: pt_faults++

  if dc_hit: dc_hits++
  else: dc_misses++

  if l2_enabled and l2_hit: l2_hits++
  else if l2_enabled: l2_misses++
```

## Programming Tips

- Update counters immediately when a decision is made
- Avoid incrementing the same counter in multiple places accidentally
- Keep statistics updates near the logic they represent
- Print the results after the entire trace is processed

## Common Mistakes

- Incrementing `mem_refs` for every cache miss instead of only when data is fetched from lower memory
- Counting an L2 hit when the L1 hit path should have finished first
- Failing to count a dirty write-back as a memory or disk reference
- Using `else if` incorrectly so one event hides another

## Verification Checklist

- TLB hit + PT not accessed
- TLB miss + PT hit
- L1 miss + L2 hit
- L1 miss + L2 miss + memory ref
- Page fault + disk ref

If you can explain where every counter increments for one reference, your simulator is on the right track.
