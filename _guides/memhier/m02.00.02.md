The simulator is mostly about storing state and updating it in the right order.

## The Right Abstraction for Each Layer

### TLB

Use an array of entries organized into sets.

Each TLB entry should contain:

- Valid bit
- Tag
- Physical page number (PPN)
- Age counter for LRU

### Page Table

Use a flat array indexed by virtual page number.

Each page table record should contain:

- Valid bit
- Physical page number (PPN)
- Dirty bit

### Data Cache and L2 Cache

Use the same structure for both caches.

Each cache line should contain:

- Valid bit
- Tag
- Dirty bit
- Age counter
- Physical page number (PPN) or other bookkeeping as needed

## Set-associative Layout

```
set 0: [way0][way1][way2]...
set 1: [way0][way1][way2]...
...
```

A cache set holds multiple candidate lines for one index.

## Why Fixed-size Arrays Help

The assignment gives maximum sizes. That means you can confidently use arrays instead of dynamic memory.

Benefits:

- Simpler initialization
- Easier to reason about
- Fewer pointer bugs

## Practical Implementation Decisions

- Use defines such as `MAX_DC_SETS`, `MAX_SET_SIZE`, `MAX_VPAGES`
- Store entries in a contiguous array
- Compute the element address with `base + set * stride + way`
- Use `unsigned` for tags and indexes

## Common Data Structure Choices

### Structs vs Raw Bytes

A small `struct` makes entries easier to read.

```c
typedef struct {
  unsigned valid;
  unsigned tag;
  unsigned dirty;
  unsigned lru;
} cache_line_t;
```

### Single Table for All Page Table Entries

A single array indexed by VPN is the cleanest design.

### Separate Caches for L1 and L2

Even if they look similar, keep separate arrays and parameters.

## Programming Tips

- Initialize the whole table with zero before use
- Keep one helper to compute set and tag from an address
- Use the same entry structure for TLB, L1, and L2 when possible
- Avoid allocating a new table inside the main loop

## Common Mistakes

- Using one cache array for both L1 and L2
- Forgetting to invalidate entries when a page is evicted
- Storing wrong type sizes for tags or page numbers
- Mixing data from different levels in one struct

## Big Picture

Design data structures so the simulator can answer questions like:

- Does this VPN already map to a PPN?
- Which cache set should this physical address use?
- Which way should be evicted next?

If you can describe the state of each layer clearly, you can implement it cleanly.
