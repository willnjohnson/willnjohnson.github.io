Bit manipulation is the bridge between the numeric address and the simulator state.

## Why Bit Math Matters

The simulator uses power-of-two sizes for a reason.

- Page sizes are powers of two
- Cache line sizes are powers of two
- Number of sets is a power of two

That means you can use shifts and masks instead of division and modulo.

## The Three Address Pieces

For a cache we need:

- `offset` = position inside the line
- `index` = which cache set
- `tag` = remaining high-order bits

### Example Calculation

Given:

- `line_size = 16`
- `nsets = 64`

Compute:

- `offset_bits = 4`
- `index_bits = 6`
- `tag_bits = address_bits - 10`

Then:

- `offset = address & (line_size - 1)`
- `index = (address >> offset_bits) & (nsets - 1)`
- `tag = address >> (offset_bits + index_bits)`

Module 1.2 covers how `vpn`, `offset`, and `physical_address` fit together. The formulas here are the same shift-and-mask trick applied to the cache side of the pipeline.

## A Worked Example

Say `dc_line_size = 16` and `num_dc_sets = 64`, so `offset_bits = 4` and `index_bits = 6`. For physical address `0x4A3`:

- `offset = 0x4A3 & 0xF = 0x3`
- `block = 0x4A3 >> 4 = 0x4A`
- `index = 0x4A % 64 = 0x0A`
- `tag = 0x4A / 64 = 0x1`

Walk through a couple of your own addresses by hand before you trust your code's output. It is much faster to catch an off-by-one in the shift amount on paper than after staring at 200 lines of wrong hit/miss output.

## Implementation Patterns

### Use Unsigned Integers

Unsigned arithmetic is safer for bit shifts, and you won't be dealing with sign extension on a right shift.

### Keep the Same Offsets for Caches and Pages

If `page_size` is 256 and `dc_line_size` is 16, you still compute the page offset from `page_size`, not from the cache line. They are unrelated quantities that happen to both be powers of two.

## Programming Tips

- Compute `offset = address & (page_size - 1)` once per reference
- Compute `vpn = address >> tlb_shift` when virtual addressing is used
- Compute cache tag/index from the physical address after translation
- Avoid `pow()` or `log2()` inside the simulation loop

## Common Mistakes

- Using `address / line_size` when you can use `>> line_bits`
- Computing `tag` from the raw trace address when physical translation is enabled
- Using the wrong mask for the number of sets
- Forgetting that `index` depends on the physical address, not the virtual address

## Why Do This

If the address math is clear, the rest of the simulator becomes bookkeeping. Division and modulo are also slower than a shift and mask, though modern compilers optimize power-of-two division for you anyway. The real payoff is correctness, not speed: a shift/mask pair only has one right answer, while `/` and `%` are easy to get subtly wrong on the wrong operand.
