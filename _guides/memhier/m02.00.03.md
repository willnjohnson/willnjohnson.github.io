Bit manipulation is the bridge between the numeric address and the simulator state.

## Why Bit Math Matters

The simulator uses power-of-two sizes for a reason.

- Page sizes are powers of two
- Cache line sizes are powers of two
- Number of sets is a power of two

That means you can use shifts and masks instead of division and modulo.

## The Three Address Pieces

For a cache we need:

- `offset` = position inside the line
- `index` = which cache set
- `tag` = remaining high-order bits

### Example Calculation

Given:

- `line_size = 16`
- `nsets = 64`

Compute:

- `offset_bits = 4`
- `index_bits = 6`
- `tag_bits = address_bits - 10`

Then:

- `offset = address & (line_size - 1)`
- `index = (address >> offset_bits) & (nsets - 1)`
- `tag = address >> (offset_bits + index_bits)`

## Virtual-to-physical Translation

A page address uses the same `offset` in both spaces.

- `vpn = address >> page_offset_bits`
- `offset = address & (page_size - 1)`

After translation:

- `physical_address = (ppn << page_offset_bits) | offset`

## Implementation Patterns

### Precompute Values from the Config

After reading the config, compute values like:

- `page_size`
- `dc_line_size`
- `l2_line_size`
- `tlb_shift = lg2(page_size)`
- `dc_shift = lg2(dc_line_size)`
- `l2_shift = lg2(l2_line_size)`

Then use these values in the main loop instead of recomputing powers of two every time.

### Use Unsigned Integers

Unsigned arithmetic is safer for bit shifts. Plus, you won't be dealing with "negative" values.

### Keep the Same Offsets for Caches and Pages

If `page_size` is 256 and `dc_line_size` is 16, you still compute the page offset from `page_size`, not from the cache line.

## Programming Tips

- Compute `offset = address & (page_size - 1)` once per reference
- Compute `vpn = address >> tlb_shift` when virtual addressing is used
- Compute cache tag/index from the physical address after translation
- Avoid `pow()` or `log2()` inside the simulation loop

## Common Mistakes

- Using `address / line_size` when you can use `>> line_bits`
- Computing `tag` from the raw trace address when physical translation is enabled
- Using the wrong mask for the number of sets
- Forgetting that `index` depends on the physical address, not the virtual address

## Why Do This

If the address math is clear, the rest of the simulator becomes bookkeeping. Plus, division is a costly operation (though compilers are optimized these days to tackle this problem).

The code should read like:

- Decode address
- Translate
- Lookup cache
- Update state

That is a reliable structure for your own implementation.
