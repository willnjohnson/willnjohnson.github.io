Good simulators follow a flow, not a single monolithic body of code.

## The Stage-based Approach

Draw your simulator as a pipeline:

```
Trace input
   |
   v
Parse access
   |
   v
Translate address
   |
   v
Lookup data cache
   |
   v
Lookup L2 cache (if enabled)
   |
   v
Load from memory / disk
   |
   v
Update statistics
```

Each item should be a function in your code.

## Why Pipelines Help

If your simulator is one giant loop body with translation, cache lookups, and statistics all tangled together, a single wrong bit shift anywhere breaks the entire output and you have no way to isolate which stage is lying to you. Splitting the pipeline into named stages gives you a seam to insert print statements, disable a layer while testing another, and compare intermediate values against the trace-by-trace breakdown you can generate with the [JS simulator](/apps/memhier).

## Example Stage Contract

Each stage should receive only the inputs it needs and return only the outputs it produces.

For example, a cache lookup stage might take:

- `access_type`
- `physical_address`
- `cache configuration`

And return:

- `hit or miss`
- `entry reference`
- `evicted line info`

## Example Pipeline Implementation

A concrete `process_reference()` function helps make the code direct and testable:

```
void process_reference(char access, unsigned long raw_address) {
  parse_trace_line(access, raw_address, &context);
  translate_address(&context);
  if (access_dc(&context)) return;
  if (l2_enabled && access_l2(&context)) {
    fill_l1_from_l2(&context);
    return;
  }
  load_from_memory(&context);
  update_statistics(&context);
}
```

This is just the shape, not the logic. `translate_address`, `access_dc`, and `access_l2` still have to do the real work described below, but resist the urge to write all of that inline in one function. That split alone will save you from most of the "works until I flip one config flag" bugs.

The translation stage should handle:

- `vpn`, `offset`, `ppn`
- TLB hit/miss
- page table hit/fault
- physical page allocation and eviction

The cache stage should handle:

- tag/index calculation
- hit detection
- replacement and dirty bits
- write-allocate vs no-write-allocate behavior

## Implementation Pattern

Use a small struct or set of variables as the reference context:

- `access_type`
- `address`
- `vpn`
- `ppn`
- `offset`
- `cache_tag`
- `cache_index`

Pass that data through the stages.

## Writes Are Not Just Reads With a Flag

The pipeline above is the same for reads and writes, but what happens *inside* each stage differs, and this is where students lose the most points. On a write:

- `L1 hit` may set dirty bit
- `L1 miss` may still avoid fill if no-write-allocate
- `L2` behavior depends on write policy

## Programming Tips

- Keep translation and cache logic in separate helper functions, even if it means passing several variables between them.
- Have each stage return a status (hit, miss, no-allocate) instead of mutating global state directly. It makes the caller's branching much easier to read back later.

## Common Mistakes

- Embedding translation logic inside cache lookup
- Updating the physical address before checking the TLB
- Making loose assumptions about layer order
- Trying to handle cache fill and page fault together in one function

## Layered Testing Strategy

Design the simulator so you can run it with only one layer active.

- First run with only DC enabled
- Then enable L2
- Then enable virtual translation
- Then enable TLB

That is the power of a pipeline model.
